## Introdução

No **Tutorial 7** estudamos a fundo a anatomia do módulo `iio_dummy`, um *driver* de exemplo para melhor entendermos como o kernel representa e gerencia dispositivos de sensores. O tutorial apresenta códigos reais desse *driver* para vermos na prática como são os conceitos ensinados.

No **Tutorial 8** colocamos a mão na massa e usamos o `iio_dummy` para aplicar tudo o que vimos até aqui.

## Minha experiência

### Tutorial 7

O tutorial propõe uma abordagem sistemática para aprender sobre qualquer *Device Driver*:

* Identificar funções de inicialização.
* Descobrir os elementos essenciais do subsistema.
* Analisar as funções de leitura e escrita
* Integrar os conceitos através da função `probe`

O conceito de canal é fundamental para o fluxo de dados no IIO, vemos como a estrutura `iio_chan_spec` cuida dos canais de dispositivos, definindo sua identifação, atributos (*masks*) e *buffers*.

A função de leitura, no nosso caso `iio_dummy_read_raw`, traduz o estado interno do sensor para o formato exigido pelo kernel, usando uma *mask* para saber o que foi pedido. Já a de escrita, `iio_dummy_write_raw`, permite a alteração de configurações e do estado do sensor.

A função `probe` une tudo o que vimos. Ela é responsável por alocar a memória, inicializar os campos do dispositivo e registrá-lo.

### Tutorial 8

Comecei com a configuração do módulo através do `nconfig`, habilitando tudo que era necessário para o funcionamento do nosso *driver*. Usei do `configfs` para gerenciar os dispositivos ativos, garantindo a limpeza dos diretórios antes do descarregamento.

Avançamos então para a modificação do código para adicionar suporte a uma bússola. Atualizei a estrutura do *driver*, defini novos canais e mapeei os índices para evitar conflitos. Por fim, o `iio_dummy_read_raw` foi adaptado para lidar com o novo canal. Após nova compilação e instalação, os testes ocorreram como esperado.
